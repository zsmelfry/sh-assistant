## 1. 产品概述

### 1.1 产品定位

**可扩展日常助手工具** 是一个个人效率工具平台，采用插件化架构，支持用户在统一界面内使用多种日常工具。所有工具共享同一数据源，工具之间可互相引用数据，实现 1+1 > 2 的协同效果。

### 1.2 产品目标

- 提供统一、简洁的工具使用入口
- 支持持续扩展新工具，无需重构已有功能
- 工具间共享数据源，降低数据孤岛风险
- 极简黑白色调，聚焦内容本身

### 1.3 目标用户

注重自我管理的个人用户，希望通过简洁的工具培养和追踪日常习惯。

### 1.4 MVP 范围

首个版本实现平台框架 + 日历打卡工具。

## 3. 功能需求

### 3.1 平台框架

#### 3.1.1 工具注册与切换

- 平台维护一个工具注册表，每个工具以独立模块形式注册
- 侧边栏展示所有已注册工具，点击切换主内容区
- 新工具只需实现约定接口即可接入平台，无需修改平台代码

#### 3.1.2 共享数据源

- 所有工具共享同一个数据存储层
- 数据层提供统一的 CRUD API，各工具通过命名空间隔离数据
- 数据持久化到本地（localStorage / IndexedDB），MVP 阶段无需后端

### 3.2 日历打卡工具

#### 3.2.1 习惯管理

- **创建习惯**: 输入习惯名称 + 选择打卡频率即可创建，名称必填，频率必选
  - 频率选项：`daily`（每天）、`weekly`（每周）、`monthly`（每月）
  - 默认频率为 `daily`
- **编辑习惯**: 可修改习惯名称和打卡频率
  - 修改频率后，已有打卡记录保留，统计按新频率重新计算
- **删除习惯**: 删除习惯时需二次确认，删除后关联打卡记录一并清除
- **习惯列表**: 展示所有习惯，支持点击切换当前查看的习惯
  - 每个习惯名称旁显示频率标签（如 `日` / `周` / `月`），便于区分

#### 3.2.2 打卡操作

**通用规则：**
- 在日历视图中，点击某天 → 完成打卡（该日期格子出现打卡标记）
- 再次点击已打卡的日期 → 取消打卡（标记消失）
- 支持对过去的日期进行补打卡
- 未来日期不可打卡

**按频率区分的打卡规则：**

| 频率 | 打卡规则 | 完成判定 |
|------|---------|---------|
| `daily` | 每天打卡一次 | 当天有打卡记录即为完成（现有逻辑不变） |
| `weekly` | 一周内（周一至周日）打卡至少一次 | 该自然周内存在至少一条打卡记录即算该周完成 |
| `monthly` | 一个月内打卡至少一次 | 该自然月内存在至少一条打卡记录即算该月完成 |

- `weekly` / `monthly` 习惯：打卡操作仍然是点击具体日期，记录打卡在哪一天完成
- 当 `weekly` 习惯本周已有打卡记录时，日历上该周其余天数显示已完成状态（浅色标识），用户仍可在其他天点击打卡（记录多次打卡）
- 当 `monthly` 习惯本月已有打卡记录时，日历上该月其余天数显示已完成状态（浅色标识），用户仍可在其他天点击打卡（记录多次打卡）

#### 3.2.3 日历视图

- 默认展示当前月份的日历
- 支持切换月份（上个月 / 下个月）
- 今天的日期有特殊高亮标识

**按频率区分的日历展示：**

| 频率 | 日历展示方式 |
|------|------------|
| `daily` | 每个日期格子独立标记打卡状态（已打卡 / 未打卡），与现有逻辑一致 |
| `weekly` | 实际打卡日显示实心标记；若该周已完成，同周其余天数显示浅色完成背景，形成**整周完成带**效果 |
| `monthly` | 实际打卡日显示实心标记；若该月已完成，日历顶部显示**本月已完成**标识（如 ✓ 徽章），其余天数显示浅色完成背景 |

- `weekly` 视图下，日历以周为视觉单元，每周一到周日为一行，周与周之间有细微间隔区分
- `monthly` 视图下，月度完成状态在日历顶部区域集中展示

#### 3.2.4 统计展示

**按频率区分的统计指标：**

| 频率 | 连续指标 | 完成率 | 年度视图（P2） |
|------|---------|--------|--------------|
| `daily` | 连续打卡**天**数（截止今天） | 本月完成率 = 已打卡天数 / 已过天数 | 年度打卡热力图（按天着色） |
| `weekly` | 连续完成**周**数（截止本周） | 本月周完成情况 = 已完成周数 / 已过周数（含本周） | 年度周完成概览（52周格子） |
| `monthly` | 连续完成**月**数（截止本月） | 本年完成情况 = 已完成月数 / 已过月数（含本月） | 年度月完成概览（12个月格子） |

- 统计区域根据当前习惯的频率自动切换展示内容
- 连续指标的中断规则：
  - `daily`：某天未打卡则中断
  - `weekly`：某个自然周内无打卡记录则中断
  - `monthly`：某个自然月内无打卡记录则中断

## 4. 交互设计要求

### 4.1 整体布局

```
+--------------------------------------------------+
|                    顶部栏 (可选)                   |
+----------+---------------------------------------+
|          |                                       |
|  侧边栏  |            主内容区                    |
|  (工具   |     (当前工具的内容)                    |
|   列表)  |                                       |
|          |                                       |
|          |                                       |
+----------+---------------------------------------+
```

- **侧边栏**: 窄栏，图标 + 工具名称，当前选中工具高亮
- **主内容区**: 展示当前选中工具的完整界面
- 侧边栏支持折叠（仅显示图标），节省空间

### 4.2 日历打卡 - 交互流程

#### 4.2.1 首次使用

1. 进入打卡工具，提示"还没有习惯，点击创建第一个"
2. 点击创建 → 弹出表单 → 输入名称 + 选择频率（默认"每天"）→ 确认
3. 自动选中刚创建的习惯，展示当月日历

#### 4.2.2 日常打卡

1. 进入打卡工具，默认选中第一个习惯，展示当月日历
2. 习惯列表中每个习惯名称旁显示频率标签（`日` / `周` / `月`）
3. 已打卡的日期显示实心圆点标记
4. 点击今天的日期 → 出现打卡标记 → 统计数字更新（天数/周数/月数视频率而定）
5. 切换习惯 → 日历刷新为该习惯的打卡数据，统计区域切换为对应频率的指标
6. `weekly` 习惯：打卡后该周整行显示完成背景
7. `monthly` 习惯：打卡后日历顶部显示本月已完成标识

#### 4.2.3 查看历史

1. 点击左右箭头切换月份
2. 日历展示对应月份的打卡情况
3. 页面顶部统计数字同步更新

### 4.3 UI 规范 - 黑白色调

| 元素 | 规范 |
|------|------|
| 背景色 | `#FFFFFF`（主区域），`#FAFAFA`（侧边栏） |
| 主文字色 | `#1A1A1A` |
| 次要文字 | `#666666` |
| 边框/分割线 | `#E5E5E5` |
| 打卡标记 | `#1A1A1A`（实心圆点） |
| 周/月完成背景 | `#F0F0F0`（浅灰色，表示该周/月已达标） |
| 频率标签 | `#999999` 文字 + `#F5F5F5` 背景，小号字体 |
| 今日高亮 | `#000000` 边框加粗 |
| 选中状态 | `#000000` 背景 + `#FFFFFF` 文字 |
| 悬停状态 | `#F5F5F5` 背景 |
| 禁用状态 | `#CCCCCC` 文字 |

**设计原则**:
- 纯黑白灰，无彩色
- 大量留白，内容优先
- 字体使用系统默认无衬线字体
- 圆角统一使用 4px（小元素）/ 8px（卡片/面板）
- 动画简洁克制，过渡时间 150ms

## 5. 数据模型需求

### 5.1 核心实体

#### Tool（工具注册表）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | string | 工具唯一标识 |
| name | string | 工具显示名称 |
| icon | string | 工具图标标识 |
| component | string | 工具组件路径/标识 |
| order | number | 排列顺序 |

#### Habit（习惯）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | string (UUID) | 习惯唯一标识 |
| name | string | 习惯名称 |
| frequency | enum: `daily` \| `weekly` \| `monthly` | 打卡频率，默认 `daily` |
| createdAt | timestamp | 创建时间 |
| updatedAt | timestamp | 更新时间 |
| archived | boolean | 是否归档（软删除） |

#### CheckIn（打卡记录）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | string (UUID) | 记录唯一标识 |
| habitId | string | 关联习惯 ID |
| date | string (YYYY-MM-DD) | 打卡日期 |
| createdAt | timestamp | 记录创建时间 |

### 5.2 实体关系

```
Tool (1) ──── 独立注册，无外键关系

Habit (1) ──< CheckIn (N)
  一个习惯对应多条打卡记录
  通过 habitId 关联
```

### 5.3 数据约束

- `CheckIn` 的 `(habitId, date)` 组合唯一，同一习惯同一天只能有一条记录
- `date` 不能大于今天
- 删除习惯时级联删除所有关联的 `CheckIn` 记录

### 5.4 共享数据层设计要点

- 统一的 Store 接口: `get(key)`, `set(key, value)`, `delete(key)`, `query(filter)`
- 各工具通过命名空间前缀隔离: `habit:xxx`, `checkin:xxx`
- 底层存储抽象化，MVP 用 localStorage/IndexedDB，后续可替换为远程 API
- 数据变更支持订阅/监听，便于 UI 响应式更新

## 6. 非功能需求

### 6.1 性能

- 首屏加载时间 < 1s（本地应用）
- 日历切换月份响应 < 100ms
- 打卡操作即时反馈，无需等待

### 6.2 可扩展性

- 新增工具只需：创建工具组件 + 在注册表中注册，无需修改平台框架代码
- 数据层设计支持未来接入后端 API，无需改动上层工具代码
- 工具组件懒加载，不影响平台启动性能

### 6.3 数据安全

- MVP 阶段数据存储在浏览器本地，需提醒用户清除浏览器数据会丢失记录
- 未来可增加数据导出/导入功能（JSON 格式）

### 6.4 兼容性

- 现代浏览器（Chrome、Firefox、Safari、Edge 最新版本）
- 响应式布局，支持桌面端和平板端

> 以下为产品侧建议，最终技术选型由架构师决定。
